Que sont les Rôles et les Actions sécurisées ?

Un rôle determine les fonctionnalités auxquelles un utilisateur peut accéder dans le Client Web. Les rôles sont des actions sécurisées qui contrôlent l'accès à des éléments de l'interface utilisateur tels que : pages, menus, éléments de menu, barres d'outils et boutons. Lorsque vous contrôlez l'accès à des éléments individualisés de l'interface utilisateur, vous pouvez contrôler l'accès à des actions tels que les modifications, les suppressions ou les impressions. Les éléments sécurisés courants sont tous des éléments de l'interface utilisateur, qui donnent accès aux comptes, aux contacts et aux opportunités.

Vous pouvez sécuriser un élément de l'interface utilisateur en affectant une action sécurisée à sa propriété Sécurité appliquée dans l'application Architect. Une fois qu'une application sécurisée est appliquée à un élément de l'interface utilisateur, seuls les utilisateurs auxquels a été attribué un rôle qui comprend l'action sécurisée peuvent voir et utiliser cet élément. Par exemple, Entities/Account/Save est l'action sécurisée affectée à la propriété Sécurité appliquée pour action du bouton Enregistrer sur la vue Détails du compte. Lorsque ce bouton se charge à l'exécution, la valeur d'affichage change selon que l'utilisateur possède ou non l'autorisation d'effectuer l'action.

Un élément de l'interface utilisateur dans le Client Web qui est sécurisé restreint l'accès à tous les utilisateurs jusqu'à ce qu'un rôle soit attribué à l'utilisateur et lui en autorise l'accès. Pour sécuriser des éléments dans le Client Web à l'aide de rôles, vous devez ajouter des actions sécurisées à ces éléments, ajouter l'action sécurisée à un rôle et attribuer ce rôle à l'utilisateur. Les éléments sont sécurisés dans l'application Architect en leur attribuant une action sécurisée ou en appliquant une sécurité. La valeur de la propriété Sécurité appliquée est le nom de l'action sécurisée. Pour sécuriser un élément dans l'IU et permettre aux utilisateurs d'accéder à cet élément, vous devez :

  1. Déterminer quelle action sécurisée vous souhaitez utiliser pour sécuriser l'élément. Si nécessaire, vous pouvez créer une nouvelle action sécurisée dans le Client Web.
  2. Ajouter une action sécurisée à la propriété Sécurité appliquée de l'élément dans l'application Architect. Pour plus d'informations, consultez la rubrique « Comprendre les actions sécurisées » dans l'aide de l'application Architect.
  3. Ajouter l'action sécurisée pour cet élément à un rôle dans le client Web.
  4. Affecter des utilisateurs au rôle.

Rôles par défaut

Par défaut, les utilisateurs créés dans le client Web se voient attribuer le rôle d'Utilisateur standard. Cela permet l'accès à toutes les fonctions non administratives dans le client Web. Si vous supprimez un utilisateur de ce rôle, l'utilisateur n'aura plus accès aux éléments sécurisés tant que vous ne lui attribuerez pas un nouveau rôle.

Les rôles par défaut sont les suivants :

  • Administrateur : permet d'ajouter, de modifier et de supprimer des autorisations à des fonctions d'administration comme les utilisateurs, les équipes, les services, les listes de sélection, les produits, les lots, la recherche de doublons, le traitement des doublons et l'historique de la recherche de doublons. Permet également de modifier les quick forms dans le concepteur de formulaire Web.
  • Responsable qualité des données : permet d'ajouter, de modifier et de supprimer des autorisations pour les fonctions Recherche des doublons, Traiter les doublons et Historique de la recherche des doublons.
  • Utilisateur standard - Vous permet de visualiser, d'ajouter, de modifier et de supprimer des autorisations aux entités non administratives du client Web telles que les comptes, les contacts, les opportunités, les lots, les produits, les commandes et les leads.
    Si vous avez des utilisateurs existants ou que vous créez de nouveaux utilisateurs dans l'Administrateur, vous devez attribuer les utilisateurs au rôle Utilisateur standard.
  • Processeur commande et devis : permet d'ajouter et d'afficher des autorisations accordées à des contacts et à des comptes, et de supprimer des autorisations pour des bons de commandes.
  • Verrouillage du prix du contrat d'intégration : permet d'accorder l'autorisation de verrouiller le prix d'un produit sur un bon de commande. (Intégration de la comptabilité uniquement)
  • La sécurité dans le client réseau est définie à l'aide de la sécurité des fonctionnalités, de la sécurité des fonctions et des rôles administratifs. La sécurité Web combine tous les éléments réseau dans un seul modèle de sécurité : un rôle. Vous ne pouvez pas utiliser le modèle de sécurité réseau pour un utilisateur qui est connecté au client Web ou le modèle de sécurité Web pour un utilisateur qui est connecté au client réseau.
  • L'utilisateur admin peut accéder à tous les éléments de l'interfacce utilisateur, pour toutes les actions sécurisées.
